# agentGroup Codex Multi-Agent Protocol

## 目标

把这个仓库作为 Codex 的多 agent 协作工作区使用。主线程是协调器，不承担所有细节工作；子代理负责边界清晰的探索、实现或验证任务。

## 角色映射

| 逻辑角色 | Codex 形态 | 何时使用 | 写权限 |
|------|------------|---------|-------|
| 主协调器 | 主线程 | 需求理解、计划、集成、最终答复 | 可写，负责最终集成 |
| 设计探索 | `explorer` | 设计分析、现状调研、方案收敛、只读审查 | 默认只读 |
| 实现执行 | `worker` | 编码、重构、补测试、修复问题 | 仅限明确 `WRITE_SCOPE` |
| 审查验证 | `explorer` 或 `worker` | 代码审查、复现问题、执行测试、补独立验证 | 默认只读；需要补测试时才写 |

## 调度原则

1. 主线程先做高层计划，再决定哪些任务值得委派。
2. 立刻阻塞主路径的工作通常由主线程直接做，不外包。
3. 子代理只接收具体、可验证、可独立完成的子任务。
4. 多个 `worker` 并行时，写入范围必须互不重叠。
5. 子代理之间不直接通信；需要共享信息时，由主线程通过 prompt 或文件中转。
6. `wait_agent` 只在主线程确实被结果阻塞时使用，避免空等。

## 六段式子任务提示

给 Codex 子代理的任务统一使用以下 6 段结构，保持可复用和可审计：

```text
PURPOSE: [最终目的]
TASK: [当前子任务，写清交付物]
MODE: [auto|write]
CONTEXT: [必须阅读的文件、目录、依赖关系]
EXPECTED: [输出结果、测试要求、返回格式]
RULES: [边界、约束、WRITE_SCOPE、禁止事项]
```

说明：

- `MODE:auto`：允许在既定范围内自主分析、实现、运行验证。
- `MODE:write`：只做目标文件的定点修改和必要验证。
- `RULES` 中必须显式写出 `WRITE_SCOPE` 和 `TEST_COMMAND`，除非任务明确只读。

## 主线程执行流程

1. 读取用户需求，识别是否需要设计、实现、审查三个工作流。
2. 本地检查代码和文档，确认依赖、风险和潜在并行机会。
3. 对每个可并行子任务定义：
   - 目标
   - 上下文输入
   - 写入范围
   - 验证方式
4. 使用 `spawn_agent` 创建子代理。
5. 主线程继续做集成准备、补充上下文或处理关键路径工作。
6. 在需要结果时使用 `wait_agent`。
7. 统一审阅改动、运行最终验证、向用户汇总结果。

## 共享产物约定

```text
.dev-agents/shared/
├── tasks/       # 子任务说明、拆解结果
├── designs/     # 设计草稿、交互说明、界面规格
├── reviews/     # 审查结论、风险列表、验收结果
└── templates/   # 复用模板
```

约束：

- 设计类输出写入 `designs/`
- 审查类输出写入 `reviews/`
- 任务拆解写入 `tasks/`
- 主线程在派发后续任务时必须引用这些产物的路径

## Persona 使用方式

`.dev-agents/ella`、`.dev-agents/jarvis`、`.dev-agents/kyle` 不是为了角色扮演，而是为了给 Codex 子代理提供：

- 输入要求
- 输出格式
- 职责边界
- 常见禁区

主线程在派发对应类型任务时，应先让子代理读取相应 `PERSONA.md`，再执行具体任务。

## 并行安全规则

1. 不把同一文件交给两个 `worker` 同时修改。
2. 审查代理默认不改代码，除非任务明确要求补测试或修复验证脚本。
3. 主线程负责最终冲突处理和结果合并。
4. 任何子代理如果发现上下文不足，应返回缺失项，而不是自行猜测。

## 最小派发模板

```text
先读取 .dev-agents/{persona}/PERSONA.md。

PURPOSE: [本轮工作的最终目的]
TASK: [你要完成的单一子任务]
MODE: auto
CONTEXT: [文件路径1, 文件路径2, 共享产物路径]
EXPECTED: [产出内容、是否改代码、需要返回哪些文件]
RULES: WRITE_SCOPE=[路径范围]; TEST_COMMAND=[命令或 N/A]; 不要修改范围外文件；完成后列出变更文件。
```

## 默认质量线

- 先对齐现有模式，再动手修改
- 只做满足当前任务所需的最小改动
- 改动后要执行对应验证
- 汇总时优先报告风险、结果和下一步
