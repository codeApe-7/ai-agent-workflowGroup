# 需求验证

对 brainstorming 阶段收集的需求进行系统性验证，确保需求完整、可行、无歧义。

```
此阶段是需求收集到方案设计之间的质量关口。
未通过验证的需求不得进入 design 阶段。
```

## 状态机

```bash
# 确认当前阶段
bash scripts/harness/workflow-state.sh gate validation
# 完成后推进到方案设计
bash scripts/harness/workflow-state.sh advance
```

## 检查清单

你**必须**按顺序完成以下每一项：

1. **读取需求文档** — 从 `.dev-agents/shared/designs/` 加载需求文档
2. **完整性检查** — 每条功能需求是否可独立理解？是否有遗漏的场景？
3. **可行性检查** — 在当前技术栈和时间约束下，每条需求是否可实现？
4. **歧义检查** — 需求是否可以被两种方式理解？如果有，选定一种并明确
5. **一致性检查** — 需求之间有没有矛盾？
6. **可验证性检查** — 每条需求是否有明确的通过/不通过标准？
7. **产出验证结论** — 在需求文档末尾追加验证结论
8. **用户确认** — 如有修改请用户确认

## 验证维度

### 完整性

- 每个用户场景是否有对应的功能需求？
- 异常情况是否被考虑？（网络错误、空数据、权限不足）
- 非功能需求是否完整？（性能、安全、兼容性）
- 范围排除是否明确？

### 可行性

- 技术上是否可实现？（不依赖不存在的 API、不违反物理限制）
- 时间上是否合理？（单次开发周期能否完成）
- 是否与现有系统兼容？（不会破坏现有功能）

### 无歧义性

- 每条需求是否只有一种理解方式？
- 数值边界是否明确？（"快速" → "200ms 内响应"）
- 行为边界是否明确？（"支持多种格式" → "支持 JSON 和 XML"）

### 一致性

- 不同需求之间是否有矛盾？
- 约束条件之间是否冲突？
- 优先级排序是否清晰？

## 验证结论格式

在需求文档末尾追加：

```markdown
---

## 需求验证结论

**验证日期**：YYYY-MM-DD
**验证结论**：通过 / 有条件通过 / 不通过

### 验证结果
- [x] 完整性：通过
- [x] 可行性：通过
- [x] 无歧义性：通过
- [x] 一致性：通过
- [x] 可验证性：通过

### 发现的问题（如有）
1. [问题描述] → [修正方案]

### 修改记录
- [修改内容]
```

## 铁律

```
1. 每条需求必须逐条验证，不可批量跳过
2. 发现问题必须现场修正，不可留到后续阶段
3. 验证结论必须写入需求文档，不可口头声明
4. validation 的唯一出口是 design（方案设计）
```

## Red Flags — 停下来

| 信号 | 行动 |
|------|------|
| 需求文档不存在 | 回到 brainstorming 阶段 |
| 需求中有"待定"或"TODO" | 现在确定，不可推迟 |
| 需求可以被两种方式理解 | 选定一种并明确 |
| 需求之间有矛盾 | 与用户澄清并修正 |
| 想跳过验证直接设计 | 不允许，验证是必须的关口 |
