# 子代理驱动开发

按实现计划逐任务派遣新的子代理（Jarvis）执行，每个任务完成后由审查子代理（Kyle）进行两阶段审查：先规格符合性，再代码质量。

**核心原则**：每任务一个新子代理 + 两阶段审查（先规格后质量）= 高质量、快迭代

**为什么用子代理**：将任务委派给隔离上下文的专门代理。通过精确构造指令和上下文，确保它们保持聚焦。子代理不应继承主会话的上下文或历史——由协调者（Max）精确构造它们所需的一切。这也为协调工作保留了主会话的上下文。

## 铁律

```
1. 每个任务一个全新子代理，禁止复用上下文
2. 两阶段审查顺序不可颠倒：先规格符合性，再代码质量
3. 审查不通过 = 修复 + 重新审查，不得跳过
4. 禁止并行派遣多个实现子代理（避免冲突）
5. 不让子代理自己读计划文件，由 Max 提取任务全文注入
```

## 流程

```
读取计划 → 提取所有任务全文 → 创建任务追踪

  ┌──────────── 每个任务循环 ────────────┐
  │                                       │
  │  派遣 Jarvis 子代理（含完整任务文本）  │
  │         ↓                             │
  │  Jarvis 实现、测试、提交、自检         │
  │         ↓                             │
  │  派遣 Kyle 子代理：规格符合性审查      │
  │    ├─ 不通过 → Jarvis 修复 → 重审     │
  │    └─ 通过 ↓                          │
  │  派遣 Kyle 子代理：代码质量审查        │
  │    ├─ 不通过 → Jarvis 修复 → 重审     │
  │    └─ 通过 ↓                          │
  │  标记任务完成                          │
  │                                       │
  └───────────────────────────────────────┘

所有任务完成 → 派遣最终整体审查 → finishing-a-development-branch
```

## 派遣 Jarvis 实现子代理

Max 在派遣 Jarvis 时必须提供：

1. **角色指令**：读取 `.dev-agents/jarvis/PERSONA.md`
2. **任务全文**：从计划中一次性提取的完整任务内容（代码、路径、命令、预期输出）
3. **上下文说明**：该任务在整体计划中的位置，与其他任务的依赖关系
4. **设计文档路径**：指向 brainstorming 产出的设计文档
5. **TDD 要求**：先写失败测试，确认失败，再写最小实现，确认通过
6. **完成标准**：实现 + 测试通过 + 自检 + 提交后报告状态

## Jarvis 状态报告

Jarvis 子代理完成后报告以下四种状态之一：

| 状态 | 含义 | Max 处理方式 |
|------|------|-------------|
| **DONE** | 完成，进入审查 | 派遣 Kyle 规格审查 |
| **DONE_WITH_CONCERNS** | 完成但有疑虑 | 先看疑虑内容，评估后再审查 |
| **NEEDS_CONTEXT** | 缺少信息 | 补充上下文，重新派遣 |
| **BLOCKED** | 无法完成 | 评估阻塞原因（见下文） |

### BLOCKED 处理

1. 如果是上下文问题 → 补充上下文，重新派遣
2. 如果任务太大 → 拆分为更小的任务
3. 如果计划本身有问题 → 上报给用户讨论
4. **永远不要**忽略阻塞或强制同一代理重试不变的内容

## 两阶段审查

### Stage 1：规格符合性审查

派遣 Kyle 子代理，提供：
- 实现计划中该任务的完整规格
- Jarvis 修改的文件路径
- 设计文档路径

Kyle 检查：
- 代码是否实现了计划的**每条**要求？
- 多做了什么？（未要求的功能 = 需要移除）
- 少做了什么？（遗漏的需求 = 需要补充）

**Stage 1 不通过** → Jarvis 修复 → Kyle 重新审查 Stage 1
**Stage 1 通过后** → 进入 Stage 2

### Stage 2：代码质量审查

派遣 Kyle 子代理，提供：
- 代码变更的 git diff
- 相关文件完整内容

Kyle 检查：
- 代码是否干净、可读、可维护？
- 有无安全隐患？
- 有无性能问题？
- 测试是否充分？
- 命名是否清晰？

**Stage 2 不通过** → Jarvis 修复 → Kyle 重新审查 Stage 2
**Stage 2 通过** → 标记任务完成，进入下一任务

## 审查报告

每次审查产出报告保存到 `.dev-agents/shared/reviews/`，包含：
- 审查阶段（规格/质量）
- 审查结论（通过/不通过）
- 具体问题清单（如有）
- 修复建议

## 整体流程完成后

所有任务完成后：
1. 派遣 Kyle 做最终整体审查（全部变更的规格 + 质量）
2. 最终审查通过 → 使用 finishing-a-development-branch 技能收尾

## Red Flags — 停下来

| 信号 | 行动 |
|------|------|
| 在 Stage 1 通过前开始 Stage 2 | 停，先完成规格审查 |
| 审查发现问题但跳过了 | 停，Jarvis 修复后重审 |
| 同时派遣多个实现子代理 | 停，一次只能一个 |
| 让子代理自己读计划文件 | 停，由 Max 提取全文注入 |
| Jarvis 报告 BLOCKED 但继续强推 | 停，评估阻塞原因 |
| 没有实现计划就开始执行 | 停，先完成 writing-plans |
| 想跳过最终整体审查 | 不允许，必须有收尾审查 |
| 质量审查在规格审查之前 | 顺序错误，先规格后质量 |

## 关联技能

- **writing-plans** — 产出实现计划（本技能的输入）
- **verification-before-completion** — Jarvis 每个任务完成时必须验证
- **systematic-debugging** — Jarvis 遇到 Bug 时使用
- **finishing-a-development-branch** — 所有任务完成后的收尾流程
