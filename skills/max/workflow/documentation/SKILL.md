# 文档更新

测试验证通过后，更新所有受影响的文档，确保文档与代码保持一致。

```
代码变更后文档不同步 = 技术债。
此阶段是消除文档债的专门时间窗口。
```

## 状态机

```bash
# 确认当前阶段
bash scripts/harness/workflow-state.sh gate documentation
# 完成后推进到分支收尾
bash scripts/harness/workflow-state.sh advance
```

## 检查清单

你**必须**按顺序完成以下每一项：

1. **列出变更文件** — 基于 git diff 和实现计划，识别本次变更涉及的所有文件
2. **识别受影响文档** — 哪些文档需要更新？
3. **逐项更新** — 按优先级更新文档
4. **一致性检查** — 确认文档与代码一致
5. **运行文档传感器** — `bash scripts/harness/lint-docs.sh`

## 文档更新优先级

### 必须更新

| 文档类型 | 触发条件 | 更新内容 |
|---------|---------|---------|
| **API 文档** | 接口变更 | 参数、返回值、示例 |
| **README** | 使用方式变更 | 安装步骤、使用示例 |
| **ARCHITECTURE.md** | 架构变更 | 组件图、数据流 |
| **模块 CLAUDE.md** | 模块行为变更 | 模块说明、约束 |

### 按需更新

| 文档类型 | 触发条件 | 更新内容 |
|---------|---------|---------|
| **代码注释** | 复杂逻辑变更 | 关键算法、设计决策 |
| **配置文档** | 配置项变更 | 配置说明、默认值 |
| **部署文档** | 部署流程变更 | 部署步骤、环境要求 |

## 更新原则

- **只更新受影响的文档** — 不做无关的文档改进
- **保持文档与代码同步** — 描述应反映实际行为
- **删除过时内容** — 不保留已删除功能的描述
- **遵循现有文档风格** — 与现有文档保持一致的格式和语气

## 执行方式

- **简单更新**：Max 直接修改
- **技术文档**：可派遣 Jarvis 更新代码相关文档
- **知识库文档**：Max 更新 docs/ 下的文档

## 铁律

```
1. 接口变更必须更新 API 文档
2. 架构变更必须更新 ARCHITECTURE.md
3. 更新完成后必须运行 lint-docs.sh 验证
4. documentation 的唯一出口是 finishing（分支收尾）
```

## Red Flags — 停下来

| 信号 | 行动 |
|------|------|
| 测试验证未通过就更新文档 | 停下来，先回到 testing 阶段 |
| 文档描述与代码不一致 | 以代码为准，修正文档 |
| lint-docs.sh 报错 | 修复后重新验证 |
| 想跳过文档更新 | 检查是否真的没有受影响文档 |
