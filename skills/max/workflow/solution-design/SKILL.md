# 方案设计

基于已验证的需求，设计技术方案和架构。产出可指导实现的技术设计文档。

```
此阶段负责"怎么做"。输入是验证过的需求文档，输出是技术设计文档。
```

## 状态机

```bash
# 确认当前阶段
bash scripts/harness/workflow-state.sh gate design
# 完成后推进到任务拆解
bash scripts/harness/workflow-state.sh advance
```

## 检查清单

你**必须**按顺序完成以下每一项：

1. **读取需求文档** — 从 `.dev-agents/shared/designs/` 加载已验证的需求文档
2. **探索现有代码** — 理解现有架构、模式、技术栈
3. **提出 2-3 种方案** — 包含取舍分析和推荐
4. **选定方案** — 用户确认后锁定
5. **详细设计** — 架构、组件、数据流、接口、错误处理
6. **撰写设计文档** — 保存到 `.dev-agents/shared/designs/YYYY-MM-DD-<功能名>-design.md`
7. **设计自检** — 检查占位符、矛盾、歧义
8. **用户确认** — 请用户审阅设计文档
9. **推进** — advance 到 planning 阶段

## 设计文档结构

```markdown
# [功能名] 技术设计文档

## 需求引用
链接到需求文档路径

## 方案对比

### 方案 A：[名称]
- 描述：...
- 优点：...
- 缺点：...

### 方案 B：[名称]
- 描述：...
- 优点：...
- 缺点：...

### 选定方案
方案 [X]，原因：...

## 架构设计
- 整体架构图
- 组件职责

## 数据流
- 输入 → 处理 → 输出

## 接口设计
- API/函数签名
- 参数和返回值

## 技术栈
- 使用的库/框架及版本

## 错误处理策略
- 异常类型和处理方式

## 测试策略
- 测试范围和方法
```

## 设计原则

- 将系统拆分为更小的单元，每个单元有一个明确目的
- 通过定义良好的接口通信，可以独立理解和测试
- 对每个单元能回答：它做什么、怎么用、依赖什么
- 外部能否不看内部就理解单元功能？能否修改内部而不破坏调用方？

## 在现有代码库中工作

- 在提议变更前先探索现有结构，遵循现有模式
- 如果现有代码有影响本次工作的问题，在设计中包含针对性改进
- 不提议无关的重构，保持聚焦

## 铁律

```
1. 必须提出至少 2 种方案并做对比分析
2. 设计文档不能有"待定""TODO"等占位符
3. 必须获得用户确认后才能推进
4. design 的唯一出口是 planning（任务拆解）
```

## Red Flags — 停下来

| 信号 | 行动 |
|------|------|
| 没有需求文档就开始设计 | 停下来，回到 brainstorming |
| 需求文档未通过验证 | 停下来，先完成 validation |
| 只提了一种方案 | 补充至少一种替代方案 |
| 设计中有"待定""之后再说" | 现在确定或标记为开放问题 |
| 想跳过用户确认 | 不允许，用户审阅是必须的门禁 |
