# 需求收集与分析

通过自然的协作对话，收集和分析功能需求，把模糊的想法转化为清晰的需求文档。

```
此阶段只负责"要做什么"，不负责"怎么做"。
技术方案设计在 solution-design 阶段完成。
```

## 状态机

```bash
# 进入此阶段
bash scripts/harness/workflow-state.sh init <任务名>
# 完成后推进到需求验证
bash scripts/harness/workflow-state.sh advance
```

## 检查清单

你**必须**按顺序完成以下每一项：

1. **探索项目上下文** — 检查文件、文档、近期提交，了解现状
2. **逐个提问澄清** — 一次一个问题，理解目的、约束、成功标准
3. **识别用户场景** — 谁会用？在什么场景下？期望什么结果？
4. **定义功能边界** — 做什么、不做什么、优先级
5. **撰写需求文档** — 保存到 `.dev-agents/shared/designs/YYYY-MM-DD-<功能名>-requirements.md`
6. **用户确认** — 请用户审阅需求文档，确认后推进到 validation

## 需求文档结构

```markdown
# [功能名] 需求文档

## 背景与目的
为什么要做这个功能？解决什么问题？

## 用户场景
- 场景 1：作为 [角色]，我希望 [行为]，以便 [价值]
- 场景 2：...

## 功能需求
- [ ] FR-1：[具体功能描述]
- [ ] FR-2：...

## 约束条件
- 性能要求：...
- 兼容性要求：...
- 安全要求：...

## 非功能需求
- ...

## 范围排除（不做什么）
- ...

## 成功标准
- ...
```

## 铁律

```
1. 此阶段不做技术方案设计，不选择技术栈，不讨论架构
2. 每条消息只问一个问题，优先使用选择题
3. 必须产出需求文档并获得用户确认才能推进
4. brainstorming 的唯一出口是 validation（需求验证）
```

## 流程详解

### 理解想法

- 先检查项目当前状态（文件、文档、近期提交）
- 如果请求涉及多个独立子系统，立即指出需要先拆解
- 逐个提问来细化想法，聚焦理解目的、约束、成功标准
- **每条消息只问一个问题**

### 定义需求

- 将用户的描述转化为结构化的功能需求列表
- 每条需求必须可验证（有明确的通过/不通过标准）
- 识别隐含需求和边界条件
- 明确范围排除（什么不做）

### 在现有代码库中工作

- 在收集需求前先探索现有结构
- 理解现有功能和约束
- 需求应考虑与现有系统的兼容性

## Red Flags — 停下来

| 信号 | 行动 |
|------|------|
| 开始讨论技术方案 | 停下来，技术方案在 design 阶段 |
| 用户说"直接做吧" | 先完成最小需求文档 |
| 需求描述了多个独立系统 | 先帮用户拆解为子项目 |
| 需求中有"待定""之后再说" | 现在就确定，或标记为明确的开放问题 |
| 想跳过用户确认 | 不允许，用户确认是必须的门禁 |

## 与团队的协作

- 如果涉及 UI/UX，可派遣艾拉参与需求收集
- 需求文档是后续所有阶段的核心输入
- 需求确认后进入 validation（需求验证）
