# 需求澄清与方案设计

通过自然的协作对话，将模糊的想法转化为完整的设计方案。

先了解项目上下文，然后逐个提问来细化想法。理解清楚要构建什么之后，分节展示设计方案并获得用户批准。

```
在展示设计并获得用户批准之前，禁止调用任何实现类技能、编写任何代码、
搭建任何项目框架或采取任何实现行动。
无论项目看起来多简单，此规则无一例外。
```

## 反模式："这个太简单了，不需要设计"

每个项目都要走这个流程。一个待办列表、一个单函数工具、一个配置变更——全都需要。
"简单"项目恰恰是未经审视的假设造成最多浪费的地方。
设计可以很短（真正简单的项目几句话就够），但你**必须**展示它并获得批准。

## 检查清单

你**必须**按顺序完成以下每一项：

1. **探索项目上下文** — 检查文件、文档、近期提交
2. **逐个提问澄清** — 一次一个问题，理解目的、约束、成功标准
3. **提出 2-3 种方案** — 包含取舍分析和你的推荐
4. **分节展示设计** — 按复杂度调节每节篇幅，每节结束后确认
5. **撰写设计文档** — 保存到 `.dev-agents/shared/designs/` 并提交
6. **设计自检** — 检查占位符、矛盾、歧义、范围（见下文）
7. **用户审阅** — 请用户审阅设计文档，确认后再继续
8. **过渡到实现** — 调用 writing-plans 技能创建实现计划

## 铁律

```
未完成设计并获得用户批准，不得调用 writing-plans 或任何实现类技能。
brainstorming 的唯一出口是 writing-plans。
```

## 流程详解

### 理解想法

- 先检查项目当前状态（文件、文档、近期提交）
- 提问前先评估范围：如果请求涉及多个独立子系统，立即指出需要先拆解，不要花时间细化一个需要分解的大项目
- 如果项目过大，帮用户分解为子项目：各部分是什么、如何关联、按什么顺序构建？然后对第一个子项目走正常设计流程
- 对合理范围的项目，逐个提问来细化想法
- 优先使用选择题，开放式问题也可以
- **每条消息只问一个问题** — 如果一个话题需要更多探索，分成多条消息
- 聚焦理解：目的、约束、成功标准

### 探索方案

- 提出 2-3 种不同方案及取舍
- 用对话方式呈现选项，附上你的推荐和理由
- 先说推荐方案，解释为什么

### 展示设计

- 确信理解了要构建什么后，展示设计
- 根据复杂度调节每节篇幅：简单的几句话，复杂的 200-300 字
- 每节展示后确认是否正确
- 覆盖：架构、组件、数据流、错误处理、测试策略
- 随时准备回头澄清不明确的地方

### 设计原则

- 将系统拆分为更小的单元，每个单元有一个明确目的，通过定义良好的接口通信，可以独立理解和测试
- 对每个单元能回答：它做什么、怎么用、依赖什么
- 外部能否不看内部就理解单元功能？能否修改内部而不破坏调用方？如果不能，边界需要重新设计

### 在现有代码库中工作

- 在提议变更前先探索现有结构，遵循现有模式
- 如果现有代码有影响本次工作的问题（如文件过大、边界不清），在设计中包含针对性改进
- 不提议无关的重构，保持聚焦

## 设计文档

### 输出

- 将验证过的设计写入 `.dev-agents/shared/designs/YYYY-MM-DD-<功能名>-design.md`
- 提交设计文档到 git

### 设计自检

写完设计文档后，用新鲜的眼光审视：

1. **占位符扫描**：有没有"待定"、"TODO"、未完成的段落、模糊的需求？修正它们
2. **内部一致性**：各节之间有没有矛盾？架构是否与功能描述一致？
3. **范围检查**：是否足够聚焦到一个实现计划？还是需要分解？
4. **歧义检查**：有没有需求可以被两种方式理解？如果有，选定一种并明确

发现问题直接修正，不需要重新审查。

### 用户审阅门禁

设计自检通过后，请用户审阅：

> "设计文档已保存到 `<路径>`。请审阅，如有修改意见请告诉我，确认后我将开始编写实现计划。"

等待用户回复。如果要求修改，修改后重新自检。**只有用户确认后才能继续。**

### 过渡到实现

- 调用 writing-plans 技能创建详细实现计划
- **不调用任何其他技能。** writing-plans 是唯一的下一步

## 核心原则

- **一次一个问题** — 不要同时抛出多个问题
- **优先选择题** — 比开放式问题更容易回答
- **严格 YAGNI** — 从所有设计中移除不必要的功能
- **探索替代方案** — 总是提出 2-3 种方案再做决定
- **增量验证** — 展示设计，获得批准后再继续
- **保持灵活** — 发现不清楚的地方随时回头澄清

## Red Flags — 停下来

发现以下情况必须停下来：

| 信号 | 行动 |
|------|------|
| 还没完成设计就想写代码 | 停下来，继续设计流程 |
| 用户说"直接做吧" | 先展示简短设计，获得确认后再继续 |
| 需求描述了多个独立系统 | 先帮用户拆解为子项目 |
| 设计中有"待定""之后再说"字样 | 现在就确定，或标记为明确的开放问题 |
| 想跳过用户审阅 | 不允许，用户审阅是必须的门禁 |
| 想调用 writing-plans 以外的技能 | brainstorming 只能接 writing-plans |

## 与团队的协作

- 如果涉及 UI/UX 设计，Max 应在 brainstorming 阶段派遣艾拉参与设计
- 设计文档是贾维斯后续开发的核心输入
- 设计完成后进入 writing-plans，再由 Max 驱动后续流水线
