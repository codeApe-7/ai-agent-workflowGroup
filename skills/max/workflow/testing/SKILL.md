# 测试验证

开发完成后，对代码进行系统性的测试验证和两阶段审查。

```
此阶段是开发到文档之间的质量关口。
未通过测试验证的代码不得进入 documentation 阶段。
```

## 状态机

```bash
# 确认当前阶段
bash scripts/harness/workflow-state.sh gate testing
# 完成后推进到文档更新
bash scripts/harness/workflow-state.sh advance
```

## 三个子步骤

### 步骤 1：测试计划编写

Max 基于实现计划（`.dev-agents/shared/tasks/`）编写或确认测试范围：

- **功能测试**：每条功能需求对应的验证用例
- **边界测试**：输入边界、空值、极端值
- **异常测试**：错误处理路径、异常恢复
- **回归测试**：确认未破坏现有功能

### 步骤 2：Stage 1 — 规格符合性审查

使用 **Agent 工具** 派遣 Kyle 子代理，提供：
- 实现计划中的完整规格
- Jarvis 修改的文件路径
- 设计文档路径

Kyle 检查：
- 代码是否实现了计划的**每条**要求？
- 多做了什么？（未要求的功能 = 需要移除）
- 少做了什么？（遗漏的需求 = 需要补充）

**Stage 1 不通过** → Agent 派遣 Jarvis 修复 → Agent 派遣 Kyle 重新审查 Stage 1

### 步骤 3：Stage 2 — 代码质量审查

Stage 1 通过后，使用 **Agent 工具** 派遣 Kyle 子代理，提供：
- 代码变更的 git diff
- 相关文件完整内容

Kyle 检查：
- 代码是否干净、可读、可维护？
- 有无安全隐患？
- 有无性能问题？
- 测试是否充分？覆盖率是否达标？
- 命名是否清晰？

**Stage 2 不通过** → Agent 派遣 Jarvis 修复 → Agent 派遣 Kyle 重新审查 Stage 2

## 审查报告格式

每次审查产出报告保存到 `.dev-agents/shared/reviews/YYYY-MM-DD-<功能名>-review.md`：

```markdown
# 审查报告：[功能名]

## 测试概要
- 测试用例总数：N
- 通过：N
- 失败：N
- 覆盖率：N%

## Stage 1：规格符合性审查
- 审查基准：[实现计划路径]
- 结论：通过 / 不通过
- 实现完整性：逐条核对结果
- 问题清单：（如有）

## Stage 2：代码质量审查
- 结论：通过 / 不通过
- 安全性：通过 / 发现问题
- 性能：通过 / 发现问题
- 可维护性：通过 / 发现问题
- 问题清单：（如有）

## 总体结论
- 状态：通过 / 不通过
- 修复建议：（如有）
```

## 铁律

```
1. Stage 1 不通过时禁止开始 Stage 2
2. 审查发现问题必须修复后重审，不得跳过
3. 审查报告必须包含完整的两阶段结构
4. testing 的唯一出口是 documentation（文档更新）
```

## Red Flags — 停下来

| 信号 | 行动 |
|------|------|
| 没有实现计划就开始审查 | 停下来，审查需要规格基准 |
| Stage 1 通过前开始 Stage 2 | 停，先完成规格审查 |
| 审查发现问题但跳过 | 停，Jarvis 修复后重审 |
| Jarvis 说"搞定了"没验证证据 | 要求运行测试并展示输出 |
| 使用"应该没问题""看起来对了" | 要求提供验证命令和输出证据 |

## 关联技能

- **subagent-driven-development** — 测试不通过时 Jarvis 修复
- **verification-before-completion** — 每次声称修复后必须验证
- **systematic-debugging** — 测试失败时的调试方法
