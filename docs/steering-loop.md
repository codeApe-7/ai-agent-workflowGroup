# 转向循环（Steering Loop）

## 核心理念

> 当一个问题反复出现时，不是更换模型或修改 prompt，而是改进 Harness——将修复编码为自动化约束，确保该问题不再发生。

## 转向循环流程

```
发现问题 → 分类问题 → 编码为约束 → 验证约束有效 → 更新质量评分
    ↑                                                    │
    └────────────── 持续监控 ←──────────────────────────┘
```

### 步骤详解

#### 1. 发现问题

问题来源：
- Kyle 审查中反复出现的同类问题
- Harness 传感器从未触发告警（可能检测不足）
- 用户反馈中的重复抱怨
- 熵管理扫描发现的漂移

#### 2. 分类问题

| 问题类型 | 编码目标 | 示例 |
|---------|---------|------|
| 结构性问题 | 计算型传感器（Linter 脚本） | 文件放错目录、缺少必要文件 |
| 规范性问题 | 前馈引导（文档/Skills 更新） | 命名不一致、注释风格错误 |
| 架构性问题 | 结构化测试 | 依赖方向违规、模块边界越界 |
| 行为性问题 | 推理型传感器（审查规则更新） | 功能逻辑错误、安全漏洞 |

#### 3. 编码为约束

根据分类选择编码方式：

- **计算型约束**（优先）：在 `scripts/harness/` 下添加或修改 Linter 脚本
  - 确保错误信息包含 `[FIX]` 修复指令
  - 确保脚本可以被 Agent 自动运行和解读

- **文档型约束**：更新 `docs/` 下的规范文档
  - 更新 `red-flags.md` 添加新的危险信号
  - 更新 `coding-standards.md` 添加新的规则

- **技能型约束**：更新 `skills/` 下的技能文件
  - 在相关 SKILL.md 中添加新的检查项

#### 4. 验证约束有效

- 运行 `scripts/harness/run-all.sh` 确认新约束被正确执行
- 制造一个违规场景，验证传感器能检测到
- 确认错误信息对 Agent 可读且包含修复指令

#### 5. 更新追踪

- 更新 `docs/QUALITY_SCORE.md` 对应维度评分
- 在 `docs/tech-debt-tracker.md` 中记录偿还
- 本文件的"改进记录"中添加条目

## 定期审查

建议每完成 5 个任务后，回顾一次：
1. 有哪些问题重复出现了？
2. 这些问题能否被编码为计算型约束？
3. 现有传感器是否有效？有无"沉默的传感器"？
