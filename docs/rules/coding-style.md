# Coding Style

## Core Principles

### KISS（保持简单）
- 用最简单可行的方案
- 避免过早优化
- 优化清晰度而非聪明

### DRY（不重复自己）
- 重复逻辑提取为公共函数
- 不要复制粘贴出现实现漂移
- 真出现重复才抽象，不要为投机抽象

### YAGNI（你不会需要它）
- 不构建当下不需要的功能
- 避免投机性通用化
- 先简单做出来，需要时再重构
- **长度压力**：能用 50 行写清楚的逻辑，不写 200 行；写完先压一遍再提交
- 不给"不可能发生的情况"加保险（如内部受控数据再做 null 校验）；边界处的校验另由 [entropy.md Golden Rule 2](entropy.md) 强制

### 防御编程
- 在系统边界（用户输入、外部 API、文件读取）做输入校验
- 用 schema 校验，快速失败并给出明确错误
- 不信任外部数据

## 动手前澄清

写代码之前的强制流程：

1. **显式声明假设** — 把"我假设 X / 我以为 Y"写出来，别藏在脑子里。前提一旦错，所有产出都白费
2. **不确定就停下，不要猜** — 命名歧义、需求模糊、行为未定义时，明确说出"哪里不清楚"，不要靠猜继续
3. **多解读时呈选项** — 任务有 2 种以上合理解读时，把候选方案列给用户选，不替用户做决定
4. **复杂方案先反问** — 方案过度设计时，先质疑能否简化；用户坚持复杂方案才执行
5. **不知道现有约定就先读** — 新模块陌生时先翻 3 个相似实现 + 对应 `docs/rules/<lang>/`，再动手

不做这一步的代价：返工、与现有模式冲突、引入与项目品味不符的代码。

## File Organization

**多个小文件 > 少个大文件**：
- 高内聚低耦合
- 200–400 行常规，800 行上限
- 大模块抽取工具函数
- 按特性/领域组织，不按类型

## Error Handling

- 每一层显式处理错误
- UI 面向用户的代码给友好消息
- 服务端记录详细上下文
- **绝不静默吞错误**

## Naming

- 变量与函数：`camelCase`，描述性命名
- 布尔值：用 `is` / `has` / `should` / `can` 前缀
- 接口、类型、组件：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 自定义 hook：`use` 前缀的 `camelCase`

## 中文注释规则

1. 所有代码注释必须使用**中文**（函数、类、模块、关键逻辑）
2. **禁止尾行注释**——注释写在代码上方单独一行
3. 一目了然的赋值与 import 语句不需要注释
4. 不写纯叙述性注释，只注释**非显而易见的意图与约束**

## Code Smells（避免）

- **深嵌套**：超过 4 层时用早返回拆开
- **魔法数字**：用具名常量代替阈值/延时/限制
- **长函数**：50 行以上拆分到职责清晰的小函数
- **变异**：优先创建新对象，不修改原对象（除非语言惯用法相反）

## 开发流程纪律

- **先读后写** — 先理解现有代码模式再动手
- **精准修改** — 只改需要改的部分
- **闭环优先** — 先跑通再优化，不过度设计
- **跟随现有风格** — 即便现有写法不是你认为的最优，也保持一致；风格统一比单点最优更重要
- **改动孤立才清理** — 只删除你的改动产生的孤立 import / 局部变量；不相关的死代码标注出来但不动手（重构走独立 PR）
- **不在功能 PR 里偷重构** — 修一个功能时顺手重构其他模块 = 评审噩梦 + 回归风险，禁止

## Code Quality Checklist

提交前自检：

- [ ] 代码可读、命名得当
- [ ] 函数小（< 50 行）
- [ ] 文件聚焦（< 800 行）
- [ ] 无深嵌套（≤ 4 层）
- [ ] 错误处理完整，边界处有校验
- [ ] 无硬编码值（用常量或 config）
- [ ] 中文注释规范，无尾行注释
- [ ] 改动范围与任务直接相关；不相关代码已标记但未顺手改
- [ ] 风格与本模块/项目现有写法一致
- [ ] 没有为不可能发生的情况加保险逻辑
- [ ] 通过 `node scripts/hooks/dispatcher.cjs stop`
