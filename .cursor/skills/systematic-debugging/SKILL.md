# 系统化调试

随机修复浪费时间并制造新 bug。快速补丁掩盖底层问题。

## 铁律

```
没有完成根因调查，不得提出任何修复方案。
```

如果还没完成第一阶段，就不能动手修。

## 何时使用

任何技术问题：测试失败、生产 bug、异常行为、性能问题、构建失败、集成问题。

**尤其是**：时间紧迫时、"一个快速修复"看起来很明显时、已经试了多次修复时。

## 四阶段流程

必须按顺序完成每个阶段。

### 第一阶段：根因调查

1. **仔细阅读错误信息** — 完整阅读堆栈跟踪，注意行号、文件路径、错误码
2. **稳定复现** — 确定可靠的触发步骤。不能复现 → 收集更多数据，不要猜
3. **检查最近变更** — git diff、最近提交、新依赖、配置变更
4. **多组件系统：在每个边界添加诊断** — 记录数据进出每个组件的状态，找到断裂点
5. **追踪数据流** — 错误值从哪里来？谁传入了错误值？一路追到源头

### 第二阶段：模式分析

1. **找到可工作的类似代码** — 代码库中有类似的正常工作的部分吗？
2. **对照参考实现** — 如果在实现某个模式，完整阅读参考实现
3. **列出差异** — 正常代码和异常代码之间有哪些不同？不要假设"这个无关紧要"
4. **理解依赖** — 需要哪些组件、配置、环境？

### 第三阶段：假设与测试

1. **形成单一假设** — 明确陈述："我认为 X 是根因，因为 Y"
2. **最小变更测试** — 一次只改一个变量
3. **验证后再继续** — 有效 → 进入第四阶段。无效 → 形成新假设，不要在上面叠加更多修复
4. **不确定时承认** — 说"我不理解 X"，不要假装知道

### 第四阶段：实现修复

1. **创建失败测试用例** — 修复前必须有
2. **实施单一修复** — 一次一个变更，不夹带"顺手改进"
3. **验证修复** — 测试通过了吗？其他测试没坏吗？
4. **修复无效 → 回到第一阶段** — 带着新信息重新分析
5. **3 次修复失败 → 质疑架构**

## 三次规则

| 修复次数 | 行动 |
|---------|------|
| < 3 | 回到第一阶段，带着新信息重新分析 |
| >= 3 | **停下来。质疑架构本身是否有问题。** 与用户讨论后再继续 |

每次修复都暴露新问题 → 这不是 bug，是架构问题。

## Red Flags — 立即停下来

如果发现自己在想：

- "先快速修一下，回头再查原因"
- "改一下 X 试试看"
- "同时改多个地方，跑一下测试"
- "跳过测试，手动验证就行"
- "大概是 X，我来修一下"
- "我不完全理解，但这个可能有用"
- "再试一次修复"（已经试了 2 次以上）

**以上全部意味着：停下来，回到第一阶段。**

## 常见借口 vs 现实

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要流程" | 简单问题也有根因。流程对简单 bug 很快 |
| "紧急情况，没时间走流程" | 系统化调试比盲猜更快 |
| "先试一下，再调查" | 第一次修复定下基调。从开头就做对 |
| "一次改多个地方省时间" | 无法隔离哪个有效。会引入新 bug |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因 |
| "再试一次"（已失败 2+ 次） | 3+ 次失败 = 架构问题。不要继续修 |
