# 需求澄清与方案设计

在动手之前，先把事情想清楚。通过对话把模糊的想法变成清晰的设计方案，获得用户批准后才进入下一步。

## 铁律

```
没有用户批准的设计方案，不得启动任何实现工作。
```

不管需求看起来多简单，都必须经过这个流程。"太简单不需要设计"是最常见的翻车原因。设计可以短（简单项目几句话），但必须呈现并获得批准。

## 何时使用

- 新功能开发
- 现有功能修改
- 架构变更
- 任何涉及多步骤的工作

**例外（Max 直接回答）**：纯知识问答、单行修改、配置调整。

## 流程清单

必须按顺序完成：

1. **探索项目上下文** — 检查现有文件、文档、最近提交
2. **逐个提问澄清需求** — 一次只问一个问题，优先提供选择题
3. **提出 2-3 种方案** — 附带权衡分析和推荐理由
4. **分段呈现设计** — 按复杂度缩放每段篇幅，每段确认后再继续
5. **写入设计文档** — 保存到 `.dev-agents/shared/designs/YYYY-MM-DD-<名称>.md`
6. **自查设计文档** — 扫描占位符、内部矛盾、模糊需求，发现就修
7. **用户审阅设计文档** — 等待用户确认后才继续
8. **转入实现计划** — 触发 writing-plans skill

## 关键原则

### 一次一个问题

不要一次抛出多个问题。如果一个话题需要深入，拆成多条消息逐个问。

### 优先选择题

能用选择题就不用开放题。开放题回答成本高，选择题更容易推进。

### YAGNI 原则

在设计阶段就砍掉不必要的功能。"以后可能用到"不是理由。

### 先探索再提问

先看项目代码和文档，再问用户。不要问项目里已有答案的问题。

### 增量验证

每段设计获得确认后再继续下一段。不要一口气写完再问。

## 设计文档格式

```markdown
# [功能名称] 设计方案

**目标**：一句话描述要构建什么
**日期**：YYYY-MM-DD
**状态**：待批准 / 已批准

## 背景与约束
[项目上下文、技术约束、用户需求]

## 方案选择
[选定的方案及理由，被否决方案的简要说明]

## 架构设计
[组件、数据流、接口定义]

## 关键决策
[重要的设计决策及其理由]
```

## 反模式：直接开干

| 借口 | 现实 |
|------|------|
| "太简单不需要设计" | 简单项目是未审视假设最容易翻车的地方 |
| "用户已经说得很清楚了" | 用户说的和你理解的往往不一样 |
| "先做个原型看看" | 没有方向的原型 = 浪费时间 |
| "需求很紧急" | 返工比设计花的时间更多 |
| "我之前做过类似的" | 每个项目的上下文不同 |

## 完成标志

设计文档已保存、用户已确认、准备进入 writing-plans 阶段。
