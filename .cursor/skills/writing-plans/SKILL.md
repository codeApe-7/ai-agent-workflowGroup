# 实现计划编写

将批准的设计方案转化为详细的实现计划。假设执行者对项目代码库零上下文、对测试设计不熟练。每个任务必须精确到文件路径、完整代码、验证命令。

## 铁律

```
计划中不得出现任何占位符。每个步骤必须包含执行者需要的全部信息。
```

## 任务粒度

每个步骤是一个 2-5 分钟的动作：

- "写失败的测试" — 一步
- "运行测试确认失败" — 一步
- "写最少代码让测试通过" — 一步
- "运行测试确认通过" — 一步
- "提交" — 一步

## 计划文档格式

每份计划必须以此头部开始：

```markdown
# [功能名称] 实现计划

**目标**：一句话描述要构建什么
**设计文档**：.dev-agents/shared/designs/xxx.md
**日期**：YYYY-MM-DD

---
```

## 任务结构

````markdown
### 任务 N：[组件名称]

**文件**：
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试确认失败**

运行：`pytest tests/path/test.py::test_name -v`
预期：FAIL，错误信息 "function not defined"

- [ ] **步骤 3：写最少实现代码**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试确认通过**

运行：`pytest tests/path/test.py::test_name -v`
预期：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: 添加具体功能"
```
````

## 禁止出现的内容

以下是计划失败的标志，绝不允许出现：

- "TBD"、"TODO"、"待定"、"后续补充"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上述代码写测试"（没有实际测试代码）
- "类似任务 N"（必须重复完整代码）
- 只描述做什么但不展示怎么做的步骤
- 引用了计划中未定义的类型、函数或方法

## 文件结构规划

在定义任务之前，先列出将创建或修改的所有文件及其职责：

- 每个文件一个清晰职责
- 优先小而聚焦的文件，不要大而全
- 在现有代码库中遵循已有模式
- 一起变更的文件应该放在一起

## 自查清单

计划写完后，用新眼光检查：

1. **需求覆盖**：设计文档中的每个需求都有对应任务吗？列出遗漏。
2. **占位符扫描**：搜索上述"禁止出现"的模式，发现就修。
3. **一致性检查**：后续任务中的类型名、方法签名和属性名与前面定义的一致吗？

发现问题就地修复，不需要重新审查。

## 保存位置

计划保存到 `.dev-agents/shared/tasks/YYYY-MM-DD-<名称>-plan.md`
