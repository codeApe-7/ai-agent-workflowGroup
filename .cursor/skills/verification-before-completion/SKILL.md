# 完成前验证

不经验证就宣称完成，是不诚实，不是高效。

## 铁律

```
没有新鲜的验证证据，不得声称任何工作已完成。
```

如果这条消息中没有运行验证命令，就不能说它通过了。

## 验证门控

在声称任何状态或表达满意之前：

1. **确定**：哪个命令能证明这个声明？
2. **运行**：执行完整命令（新鲜的、完整的）
3. **阅读**：完整输出，检查退出码，计算失败数
4. **核实**：输出是否确认了声明？
   - 否 → 如实报告实际状态和证据
   - 是 → 带着证据声明结果
5. **然后才能**：做出声明

跳过任何步骤 = 没有验证。

## 常见场景

| 声明 | 需要的证据 | 不够的证据 |
|------|-----------|-----------|
| 测试通过 | 测试命令输出：0 失败 | 之前的运行结果、"应该通过" |
| 构建成功 | 构建命令：exit 0 | lint 通过、"日志看起来没问题" |
| Bug 已修复 | 测试原始症状：通过 | 代码改了、"应该修好了" |
| 需求已满足 | 逐条检查清单 | 测试通过（测试不等于需求） |
| Agent 已完成 | VCS diff 显示变更 | Agent 报告"成功" |

## Red Flags — 立即停下来

发现自己在说或想：

- 使用"应该""大概""似乎""看起来"
- 在验证前表达满意（"搞定了！""完美！""没问题！"）
- 准备提交/推送但没有运行验证
- 信任 Agent 的成功报告而没有独立验证
- 依赖部分验证
- 想着"就这一次"
- 已经累了想快点结束

**以上全部意味着：停下来，运行验证命令。**

## 常见借口 vs 现实

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 运行验证命令 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Lint 通过了" | Lint ≠ 编译 ≠ 测试 |
| "Agent 说成功了" | 独立验证 |
| "我累了" | 疲劳 ≠ 免责 |
| "部分检查够了" | 部分证明不了整体 |

## 正确 vs 错误示例

**测试**：
```
正确：[运行测试命令] → [看到：34/34 通过] → "所有测试通过"
错误："应该通过了" / "看起来没问题"
```

**构建**：
```
正确：[运行构建] → [看到：exit 0] → "构建成功"
错误："Lint 通过了"（lint 不检查编译）
```

**需求验收**：
```
正确：重新阅读计划 → 创建检查清单 → 逐条验证 → 报告差距或完成
错误："测试通过了，阶段完成"
```

## 底线

运行命令。阅读输出。然后才能声称结果。

没有捷径。不可协商。
