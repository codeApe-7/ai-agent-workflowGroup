## TDD 铁律

```
没有先失败的测试，不得写任何生产代码。
```

### Red-Green-Refactor 循环

1. **RED** — 写一个失败的测试，描述期望行为
2. **验证 RED** — 运行测试，确认它因为功能缺失而失败（不是因为语法错误）
3. **GREEN** — 写最少代码让测试通过，不多不少
4. **验证 GREEN** — 运行测试，确认通过且其他测试没坏
5. **REFACTOR** — 在测试保持绿色的前提下清理代码
6. **重复** — 下一个失败测试

### 先写代码再补测试？删除重来。

- 不保留作为"参考"
- 不"调整"着写测试
- 不看着它写
- 删除就是删除，从测试开始重新实现

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会坏。测试只需 30 秒 |
| "回头再补测试" | 后补的测试立即通过，证明不了任何东西 |
| "TDD 会拖慢我" | TDD 比调试更快。务实 = 先测试 |
| "先探索一下" | 可以。然后扔掉探索代码，从 TDD 开始 |
| "测试很难写" | 听测试的声音。难测试 = 难使用 = 设计有问题 |
| "现有代码没测试" | 你在改进它。为你改动的部分加测试 |

## 验证铁律

```
没有运行验证命令并展示输出，不得声称任何工作已完成。
```

完成任何功能后必须：

1. 运行全部相关测试并贴出完整输出
2. 确认 0 失败、0 错误、0 警告
3. 然后才能说"完成"

禁止使用的表述：
- "应该没问题了"
- "看起来对了"
- "我很有信心"
- "大概通过了"

必须使用的表述：
- "运行 `xxx` 命令，结果：N/N 通过，0 失败"

## 调试铁律

```
遇到 bug 时，必须先完成根因调查，再提出修复方案。禁止盲猜。
```

遇到测试失败或异常行为时：

1. 完整阅读错误信息和堆栈跟踪
2. 稳定复现问题
3. 检查最近变更
4. 追踪数据流找到根因
5. 形成假设，最小变更测试
6. 3 次修复失败 → 停下来，质疑架构，请求帮助

详细流程参考 `systematic-debugging` skill。

## 开发规范

- 先读取项目已有代码，理解现有模式再动手
- 精准修改，只改需要改的部分
- 先跑通闭环再优化，不过度设计
- 代码需有清晰结构和必要的中文注释
- 遵循项目既有的代码风格和技术栈
- 考虑边界情况和错误处理
- 完成功能后主动说明实现要点

## Red Flags — 立即停下来

发现自己在做以下事情时，必须停下来纠正：

| 信号 | 正确做法 |
|------|---------|
| 先写代码再补测试 | 删除代码，从测试开始 |
| 测试一写就通过了 | 你在测试已有行为，修改测试 |
| 没看到测试失败就开始写实现 | 回去运行测试，确认失败 |
| 说"搞定了"但没运行测试 | 运行测试，贴出输出 |
| 遇到 bug 直接改代码试试 | 停下来，走系统化调试流程 |
| 一次改多个地方看效果 | 一次只改一个变量 |
| 跳过实现计划直接开干 | 等 Max 提供计划再开始 |
