# 测试驱动开发 (TDD)

Jarvis 实施代码时必须遵循的开发纪律。

## 铁律

```
没有先写失败测试，就不能写产品代码。
```

先写了代码再补测试？**删掉代码。从头开始。**

- 不保留作"参考"
- 不"改改"现有代码
- 不看已写的代码
- 删除就是删除

从测试开始，全新实现。

## 适用范围

**始终使用：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**例外（需用户明确许可）：**
- 一次性原型
- 生成的代码
- 配置文件

## 红-绿-重构循环

### RED — 编写失败测试

编写一个最小测试，展示期望行为。

**要求：**
- 测试一个行为
- 清晰的测试名称
- 使用真实代码（非必要不用 mock）

### 验证 RED — 观察测试失败

**必须执行，不可跳过。**

确认：
- 测试失败（不是报错）
- 失败信息符合预期
- 因为功能缺失而失败（不是拼写错误）

**测试通过了？** 你在测试已有行为。修改测试。

### GREEN — 最小实现

编写最简单的代码让测试通过。

不要添加功能、重构其他代码、或在测试要求之外"改进"。

### 验证 GREEN — 观察测试通过

**必须执行。**

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误、无警告）

**测试失败？** 修改代码，不是修改测试。

### REFACTOR — 清理

仅在 GREEN 之后：
- 消除重复
- 改善命名
- 提取辅助函数

保持测试通过。不添加新行为。

### 循环

下一个失败测试，下一个功能。

## 常见借口与现实

| 借口 | 现实 |
|------|------|
| "太简单不需要测试" | 简单代码也会出 Bug，测试只需 30 秒 |
| "之后再补测试" | 测试直接通过什么也证明不了 |
| "我已经手动测过了" | 临时测试 ≠ 系统测试。无记录，无法重复 |
| "删掉 X 小时的工作太浪费" | 沉没成本谬误。保留无法信任的代码才是技术债 |
| "保留作参考" | 你会适配它。那就是事后补测试。删除就是删除 |
| "需要先探索" | 可以。探索完扔掉，从 TDD 开始 |
| "测试太难写 = 设计不清楚" | 听测试的话。难测试 = 难使用 |
| "TDD 会拖慢我" | TDD 比调试快。务实 = 测试先行 |

## 停止信号 — 从头开始

- 先写代码再写测试
- 测试直接通过
- 不知道测试为什么失败
- 在合理化"就这一次"
- "我已经手动测过了"
- "保留作参考"

**以上任何一条意味着：删除代码，从 TDD 重新开始。**

## 完成检查清单

标记工作完成前：

- [ ] 每个新函数/方法都有测试
- [ ] 观察了每个测试失败
- [ ] 每个测试因预期原因失败（功能缺失，不是拼写错误）
- [ ] 为每个测试写了最小实现
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）
- [ ] 测试使用真实代码（非必要不用 mock）
- [ ] 覆盖了边界情况和错误情况
