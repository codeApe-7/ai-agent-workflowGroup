# 完成前验证

声称工作完成但没有验证，是不诚实，不是高效。

## 铁律

```
没有新鲜的验证证据，就不能声称完成。
```

如果你在当前消息中没有运行过验证命令，就不能声称它通过了。

## 验证门函数

```
在声称任何状态或表达满意之前：

1. 确定：什么命令能证明这个声明？
2. 执行：运行完整命令（新鲜的、完整的）
3. 阅读：完整输出，检查退出码，统计失败数
4. 核实：输出是否确认了声明？
   - 否 → 说明实际状态并附证据
   - 是 → 声明结果并附证据
5. 此时才能：做出声明

跳过任何步骤 = 说谎，不是验证
```

## 常见声明与所需证据

| 声明 | 需要 | 不够格 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 failures | 上一次运行、"应该通过" |
| Lint 干净 | Lint 输出：0 errors | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Lint 通过、日志看着正常 |
| Bug 已修复 | 测试原始症状：通过 | 代码改了、假设修好了 |
| 回归测试有效 | 红-绿循环已验证 | 测试通过一次 |
| 子代理完成 | VCS diff 显示变更 | 子代理报告"成功" |
| 需求满足 | 逐条检查清单 | 测试通过 |

## 停止信号

- 使用"应该"、"大概"、"看起来"
- 验证前表达满意（"太好了！"、"完美！"、"搞定！"）
- 准备提交/推送/PR 但没验证
- 信任子代理的成功报告
- 依赖部分验证
- 想着"就这一次"
- 疲倦想收工
- **任何暗示成功但没有运行过验证的措辞**

## 借口与现实

| 借口 | 现实 |
|------|------|
| "应该没问题了" | 运行验证 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Lint 通过了" | Lint ≠ 编译器 |
| "子代理说成功了" | 独立验证 |
| "我累了" | 疲倦 ≠ 借口 |
| "部分检查够了" | 部分证明不了什么 |

## 核心规则

运行命令。阅读输出。**然后**声称结果。

不可商量。
