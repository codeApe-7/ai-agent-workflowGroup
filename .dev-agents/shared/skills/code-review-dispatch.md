# 代码审查派发

Kyle 执行的结构化审查流程。每次审查分两个阶段：规格合规审查和代码质量审查。

## 触发条件

**必须审查：**
- Jarvis 完成每个任务后
- 重大功能完成后
- 合并到主分支前

**建议审查：**
- 卡住时（换个视角）
- 重构前（基线检查）
- 修复复杂 Bug 后

## 两阶段审查

### 阶段 1: 规格合规审查

**目标：** 代码是否实现了规格要求的，且仅实现了规格要求的。

检查清单：

- [ ] 每个规格需求都有对应实现
- [ ] 没有规格之外的额外功能（YAGNI）
- [ ] 接口、类型、方法签名与规格一致
- [ ] 测试覆盖了规格中的每个行为
- [ ] 错误处理覆盖了规格中定义的异常情况

**结果格式：**

```markdown
## 规格合规审查

### ✅ 通过项
- [列出符合规格的实现]

### ❌ 缺失项
- [规格要求但未实现的功能]

### ⚠️ 额外项
- [规格未要求的额外功能]

### 结论
- [ ] 规格合规 / [ ] 需要修改
```

**规格合规必须通过后，才能进入阶段 2。**

### 阶段 2: 代码质量审查

**目标：** 代码是否写得好。

检查维度：

| 维度 | 检查点 |
|------|--------|
| 正确性 | 逻辑正确、边界处理、并发安全 |
| 可维护性 | 命名清晰、职责单一、耦合度低 |
| 测试质量 | 测试真实行为而非 mock、覆盖边界 |
| 安全性 | 输入验证、注入防护、敏感信息 |
| 性能 | 无明显瓶颈、合理的复杂度 |

**结果格式：**

```markdown
## 代码质量审查

### 优点
- [列出做得好的方面]

### 问题

#### 阻断 (必须修)
- [文件:行号] 问题描述

#### 高 (应该修)
- [文件:行号] 问题描述

#### 中 (建议修)
- [文件:行号] 问题描述

#### 低 (可选)
- [文件:行号] 问题描述

### 结论
- [ ] 通过 / [ ] 需要修改
```

## 审查循环

```
Jarvis 完成任务
    ↓
Kyle 阶段 1 审查（规格合规）
    ↓
不通过 → Jarvis 修复 → Kyle 重新阶段 1
    ↓ 通过
Kyle 阶段 2 审查（代码质量）
    ↓
不通过 → Jarvis 修复 → Kyle 重新阶段 2
    ↓ 通过
标记任务完成
```

## 禁止事项

- 因为"很简单"就跳过审查
- 忽略阻断级问题
- 阶段 1 未通过就开始阶段 2
- 在有未修复的高级问题时继续下一个任务
- 用感觉代替证据给"通过"
