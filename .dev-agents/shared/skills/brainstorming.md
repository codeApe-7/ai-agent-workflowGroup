# 需求澄清与方案设计

Max 主导的协作式需求探索技能。在写任何代码、做任何设计之前，先把事情想清楚。

> **铁律：不论需求看起来多简单，都必须先产出设计方案并获得用户确认，才能进入实施。**

## 触发条件

- 用户提出新功能需求
- 用户描述一个要解决的问题
- 任何涉及「创建、构建、添加、修改行为」的请求

## 检查清单

Max 必须按顺序完成以下步骤：

1. **探索项目上下文** — 检查现有文件、文档、最近提交
2. **逐个提出澄清问题** — 每次只问一个问题，优先使用选择题
3. **提出 2-3 种方案** — 附带各自的取舍分析和推荐理由
4. **分段呈现设计** — 按复杂度分段展示，每段获得用户确认
5. **编写设计文档** — 保存至 `.dev-agents/shared/designs/T-{NNN}-<slug>.md`
6. **设计自审** — 检查占位符、内部矛盾、模糊需求、范围蔓延
7. **用户确认设计文档** — 等待用户明确批准
8. **转入计划编写** — 使用 `writing-plans` 技能创建实施计划

## 反模式：「这太简单了不需要设计」

每个项目都要走这个流程。一个配置修改、一个工具函数、一个 Todo 列表——全部。"简单"的项目正是未经审视的假设最容易浪费工作量的地方。设计可以很短（简单项目几句话就够），但必须呈现并获得批准。

## 澄清过程

### 理解需求

- 先检查当前项目状态（文件、文档、最近提交）
- 在深入细节前先评估范围：如果需求描述了多个独立子系统，立即标记并建议分解
- 逐个提问，每次一个问题
- 优先使用选择题，开放式问题也可以
- 聚焦于：目的、约束、成功标准

### 探索方案

- 提出 2-3 种不同方案及其取舍
- 以对话方式呈现，附上推荐和理由
- 推荐方案放在最前面

### 呈现设计

- 确信理解了要构建什么后，分段展示设计
- 每个部分的详细程度与其复杂度匹配：直观的部分几句话，复杂的可以 200-300 字
- 每段之后询问是否正确
- 覆盖：架构、组件、数据流、错误处理、测试策略

### 设计隔离与清晰度

- 将系统拆分为更小的单元，每个单元有一个明确目的
- 通过定义清晰的接口通信，可以独立理解和测试
- 对每个单元能回答：它做什么、怎么用、依赖什么

### 在现有代码库中工作

- 先探索现有结构再提方案，遵循已有模式
- 现有代码的问题如果影响当前工作，纳入设计改进范围
- 不做无关重构

## 设计自审

写完设计文档后，以新鲜眼光审视：

1. **占位符扫描** — 有 "TBD"、"TODO"、未完成的部分吗？修掉
2. **内部一致性** — 各部分之间有矛盾吗？架构与功能描述匹配吗？
3. **范围检查** — 是否聚焦到可以用一个实施计划覆盖？需要分解吗？
4. **歧义检查** — 有没有可以两种理解的需求？选一个，明确写出来

## 向实施过渡

设计确认后：

1. 使用 `writing-plans` 技能编写实施计划
2. 不直接调用其他技能或开始写代码
3. 实施计划是下一步，且是唯一的下一步

## 核心原则

- **一次一个问题** — 不要用大量问题淹没用户
- **选择题优先** — 比开放式问题更容易回答
- **YAGNI** — 从所有设计中无情删除不必要的功能
- **探索替代方案** — 始终提出 2-3 种方案再做决定
- **增量验证** — 呈现设计，获得批准后再继续
