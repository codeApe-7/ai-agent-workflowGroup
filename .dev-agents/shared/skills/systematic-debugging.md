# 系统化调试

随机修复浪费时间并引入新 Bug。快速补丁掩盖深层问题。

## 铁律

```
没有完成根因调查，就不能提出修复方案。
```

没完成第一阶段，不允许动代码。

## 适用范围

- 测试失败
- 生产 Bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在这些时候：**
- 时间紧迫（紧急状况让猜测变得诱人）
- "一个快速修复"看起来很明显
- 已经尝试过多个修复
- 上一个修复没有效果

## 四阶段流程

### 阶段 1: 根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 能否可靠触发？
   - 精确步骤是什么？
   - 每次都发生吗？
   - 无法复现 → 收集更多数据，不要猜测

3. **检查最近变更**
   - 什么变更可能导致此问题？
   - Git diff、最近提交
   - 新依赖、配置变更

4. **多组件系统中收集证据**
   - 在每个组件边界添加诊断日志
   - 记录数据进入/离开每个组件的状态
   - 运行一次收集证据，确定断裂层
   - 然后针对特定组件调查

5. **追踪数据流**
   - 坏值从哪里来？
   - 谁用坏值调用了这里？
   - 一直追踪到源头
   - 在源头修复，不在症状处修

### 阶段 2: 模式分析

1. **找到正常工作的示例** — 类似但正常的代码在哪？
2. **对比参考** — 完整阅读参考实现，不要略读
3. **识别差异** — 列出正常与异常之间的每个差异
4. **理解依赖** — 需要什么组件、配置、环境？

### 阶段 3: 假设与测试

1. **形成单一假设** — "我认为 X 是根因，因为 Y"
2. **最小测试** — 做最小可能的变更来测试假设
3. **验证后继续** — 有效 → 阶段 4；无效 → 新假设
4. **不在旧修复上叠加新修复**

### 阶段 4: 实施

1. **创建失败测试** — 最简复现，使用 `tdd` 技能
2. **实施单一修复** — 修复根因，不是症状。一次一个变更
3. **验证修复** — 测试通过？没破坏其他测试？
4. **如果修复无效** — 停止。已尝试多少次？
   - < 3 次：回到阶段 1，用新信息重新分析
   - ≥ 3 次：**停止，质疑架构**

### 3 次修复失败 → 质疑架构

**模式指示架构问题：**
- 每次修复暴露新的共享状态/耦合
- 修复需要"大规模重构"
- 每次修复在别处产生新症状

**停止并质疑根本问题，上报 Max 讨论。**

## 停止信号

如果你发现自己在想：
- "先快速修一下，以后再查"
- "改个 X 试试看行不行"
- "一次改多处，跑测试"
- "跳过测试，手动验一下"
- "大概是 X，修了"
- "不完全明白但也许能行"
- "再试一次修复"（已经试了 2+ 次）

**以上都意味着：停止。回到阶段 1。**

## 常见借口

| 借口 | 现实 |
|------|------|
| "问题很简单，不需要走流程" | 简单问题也有根因。流程对简单 Bug 很快 |
| "紧急情况，没时间走流程" | 系统化调试比碰运气式修复更快 |
| "先试这个，然后再调查" | 第一个修复定了模式。从一开始就做对 |
| "改多处省时间" | 无法隔离哪个有效。可能引入新 Bug |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因 |
