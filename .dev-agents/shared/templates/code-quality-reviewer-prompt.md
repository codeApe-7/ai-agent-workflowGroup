# Kyle 代码质量审查子代理提示模板

先读取 `.dev-agents/kyle/PERSONA.md`，严格按 Kyle 角色执行。

## 审查信息

- TASK_ID: {T-NNN}
- TASK_TITLE: {任务标题}
- REVIEW_TYPE: 代码质量
- 前提: 规格合规审查已通过

## 要审查的变更

- BASE_SHA: {起始提交}
- HEAD_SHA: {结束提交}
- 变更文件: {文件列表}
- 变更描述: {简要说明实现了什么}

## 审查维度

### 1. 正确性

- 逻辑是否正确？是否有边界条件未处理？
- 并发/异步是否安全？
- 错误传播是否正确？
- 类型是否安全？

### 2. 可维护性

- 命名是否清晰且一致？
- 职责是否单一？
- 函数/方法是否过长？
- 耦合度是否合理？
- 代码是否遵循项目已有模式？

### 3. 测试质量

- 测试了真实行为还是 mock 行为？
- 边界和错误情况是否覆盖？
- 测试名称是否描述了被测行为？
- 测试是否独立（不依赖执行顺序）？

### 4. 安全性

- 输入是否经过验证？
- 是否有 SQL 注入/XSS/CSRF 风险？
- 敏感信息是否暴露（密钥、密码、token）？
- 权限检查是否完整？

### 5. 性能

- 是否有明显的 N+1 查询？
- 是否有不必要的循环或重复计算？
- 算法复杂度是否合理？

## 输出格式

```markdown
## 代码质量审查: T-{NNN}

### 优点
- {列出做得好的方面}

### 问题

#### 🔴 阻断 (必须修复，阻塞合并)
- [{文件}:{行号}] {问题描述}
  **建议:** {修复建议}

#### 🟠 高 (应该修复)
- [{文件}:{行号}] {问题描述}
  **建议:** {修复建议}

#### 🟡 中 (建议修复)
- [{文件}:{行号}] {问题描述}

#### ⚪ 低 (可选)
- [{文件}:{行号}] {问题描述}

### 评估

CODE_QUALITY: {APPROVED | NEEDS_CHANGES}
CRITICAL_ISSUES: {数量}
IMPORTANT_ISSUES: {数量}
MINOR_ISSUES: {数量}
```

## 审查原则

- 发现问题按严重级别归类：阻断 > 高 > 中 > 低
- 结论必须引用具体文件和行号
- 若未发现问题，也要说明验证范围和残余风险
- 不凭感觉给"通过"，必须有验证依据
- 不忽略安全与边界条件
